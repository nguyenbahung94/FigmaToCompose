# figma-to-compose Public Plugin — Design Spec

**Date:** 2026-04-17
**Status:** Approved

---

## 0. Core Philosophy

> **This is NOT a UI generator. It is a Design → System → Code pipeline.**

```
Figma          → input, NOT source of truth
Design System  → source of truth (token-mapping.json + component-mapping.json)
Generated Code → output constrained by the system, not by Figma directly
```

> **IR = Intent tree, not UI tree.**
>
> "Layout intent: stack children vertically with spacing X" — not "Column contains children."
> Platform renders differently, but intent is the same. The IR captures intent.
> This is why there is no `dp`, no `Modifier`, no `Composable` in the IR.

If Figma uses a color not in `token-mapping.json` → FAIL (never emit hardcoded values).
If Figma uses a component not in `component-mapping.json` → FallbackLevel determines behavior.
Figma is never allowed to override the Design System. The system wins.

## 1. Target Audience

Mobile teams using Jetpack Compose (Android), with extensibility for iOS/SwiftUI in a future version. The plugin is designed for any team — not tied to any specific project or design system.

**Multi-skill, single plugin** — one install, clean platform separation:

```
figma-to-compose/                    ← GitHub repo + agentskills.io plugin
  plugin.json
  skills/
    figma-to-compose/               ← Core: Stages 1–5 (Fetch → IR). Platform-agnostic.
      SKILL.md
    figma-to-compose-android/       ← Android Stage 6 generator. Reads IR, emits Compose.
      SKILL.md
    figma-to-compose-ios/           ← iOS placeholder (v2). Stub only.
      SKILL.md
  templates/
    android/
      token-mapping.template.json
      component-mapping.template.json
  schemas/
    config.schema.json
  docs/
    specs/
    plans/
```

**Skill dependency graph:**
- `figma-to-compose` (core) — no dependencies
- `figma-to-compose-android` — depends on `figma-to-compose`
- `figma-to-compose-ios` — depends on `figma-to-compose`

Adding a new platform = add one new skill file. Zero changes to core.

## 3. Project Config Structure

After running `/figma-to-compose init`, project-local config lives at:

```
<your-project>/
  .claude/figma-to-compose/
    config.json                     ← platform, design system pointers
    mappings/
      token-mapping.json            ← generated by init, edited by team
      component-mapping.json        ← generated by bootstrap, confirmed by team
```

**config.json schema (v1.0):**
```json
{
  "platform": "android",
  "mapping_version": "1.0",
  "plugin_version": "1.x",
  "colors_source": "src/main/res/values/colors.xml",
  "typography_class": "com.example.AppTypography",
  "compose_package": "com.example.presentation"
}
```

**Field resolution rules:**
- `compose_package` = Java/Kotlin package name (e.g. `com.example.presentation`).
  Plugin resolves to output path by: replace `.` with `/`, prepend source root (e.g. `src/main/java/`).
  Result: `src/main/java/com/example/presentation/`
- `colors_source` = relative path from project root to colors resource file.
```

Plugin reads project-local config first. Plugin defaults (templates) are fallback only.

## 4. Command Set

### Setup (run in order)

| Command | Purpose |
|---|---|
| `/figma-to-compose init` | First-time setup: scan design system, generate config + token-mapping |
| `/figma-to-compose bootstrap` | Scan codebase @Composable functions, seed component-mapping. Requires init. |
| `/figma-to-compose validate` | Verify config + mappings valid before first use |

### Debug

| Command | Purpose |
|---|---|
| `/figma-to-compose inspect <url>` | Show IR tree + token resolution + mapping decisions. No codegen. |
| `/figma-to-compose <url> --layout-map` | Spatial ASCII tree (Stage 1 only). No IR. |
| `/figma-to-compose review-mappings` | Review + confirm pending component mappings |
| `/figma-to-compose suggest-mapping` | List unmapped Figma components with suggestions |
| `/figma-to-compose cleanup-orphans` | List orphan generated blocks (Figma node removed) |

### Execution

| Command | Behavior |
|---|---|
| `/figma-to-compose run <url>` | **Smart**: respects confidence gate, may auto dry-run |
| `/figma-to-compose generate <url>` | **Dumb**: always generates, no decision logic |
| `/figma-to-compose ci <url>` | **CI mode**: deterministic, fail-fast, no prompts, machine-readable output |

**Flags (run/generate only):**
- `--dry-run` — full pipeline, print IR, no file writes
- `--allow-write-low-confidence` — bypass auto dry-run gate ONLY. Does NOT bypass unresolved tokens or Major fallbacks.
- `--verbose` — show INFO logs

**CI mode rules:**
- Deterministic (same as `generate`, no confidence decisions)
- Fail-fast: exit non-zero on ANY unresolved token, Major fallback, or WARN-level issue
- No staged mode, no user prompts
- Emits machine-readable JSON report to stdout
- Intended for pre-merge gates, never for interactive use

**Rule: `run` = smart. `generate` = deterministic. `ci` = strict + machine-readable. Never mix their behaviors.**

## 5. Init Wizard Design

### Platform detection (defensive, multi-module aware)

```
1. Search ANY module for "@Composable" import OR "androidx.compose" dependency → Android
2. Multiple modules found → pick module with most @Composable functions
3. Fallback → ask user
```

Do NOT assume root project. Works with multi-module repos.

### Token scanning (multi-source, no naming assumptions)

```
colors:
  1. Parse colors_source (e.g. colors.xml) for <color name="xxx">#HEX</color>
  2. Search Color.kt / theme files for Color(0x...) definitions
  3. Fallback → ask user

typography:
  1. Search for TextStyle definitions in source
  2. Search for MaterialTheme.typography usages
  3. Fallback → ask user to select file

spacing:
  1. Parse dimens.xml
  2. Search for *.dp constants in source
  3. Fallback → ask user
```

### Component scanning (filtered)

Include: `public` / `internal` @Composable, has ≥2 params, name does NOT end with "Preview".
Exclude: `@Preview`, `private`, functions with <2 params (likely helpers).

### AI matching (score-gated)

```
score = 0.5 * name_similarity + 0.3 * param_similarity + 0.2 * usage_frequency
Accept only if score ≥ 0.70. Otherwise → mark "unmapped".
```

Never force a match. Unmapped is safer than wrong.

### Init output safety

- Generate files with `"needs_review": true` on uncertain entries
- Do NOT auto-apply mappings
- Force `/figma-to-compose validate` before first use

## 6. IR Format (Platform-Agnostic)

The IR is an **intent tree**. No platform concepts (no `dp`, `Modifier`, `Composable`, `Column`, `Row`).
IR describes **what** the layout intends — not **how** any platform renders it.
Platform skills translate IR → platform code.

### Axis Semantics (not layout system semantics)

Instead of `Vertical`/`Horizontal` (which reflect Compose/Flexbox thinking), IR uses:
- `direction` — which axis children flow along
- `mainAxis` — arrangement along the flow direction
- `crossAxis` — alignment perpendicular to flow

This maps cleanly to any platform without bias:

| Intent | Android | iOS | Web |
|---|---|---|---|
| `direction=Vertical, mainAxis=SpaceBetween, crossAxis=Start` | `Column(verticalArrangement=SpaceBetween, horizontalAlignment=Start)` | `VStack(alignment:.leading) + Spacer()` | `flex-direction:column; justify-content:space-between; align-items:flex-start` |

```kotlin
enum class Direction { Vertical, Horizontal, Stack }

enum class MainAxisArrangement {
    Start, Center, End, SpaceBetween, SpaceAround, SpaceEvenly
}
enum class CrossAxisAlignment {
    Start, Center, End, Stretch
}

data class CommonProps(
    val padding: PaddingTokens? = null,
    val background: TokenRef? = null,
    val cornerRadius: TokenRef? = null,
    val border: Border? = null,
    val action: Action? = null,
)

data class SlotValue(
    val nodes: List<LayoutNode>,
    val required: Boolean = false,
)

sealed class LayoutNode {
    data class Layout(
        val direction: Direction,
        val common: CommonProps,
        val spacing: TokenRef,
        val mainAxis: MainAxisArrangement = MainAxisArrangement.Start,
        val crossAxis: CrossAxisAlignment = CrossAxisAlignment.Start,
        val width: SizeMode,
        val height: SizeMode,
        val children: List<LayoutNode>,
    )
    data class Component(
        val mappingKey: String,
        val props: Map<String, PropValue>,
        val slots: Map<String, SlotValue>,
    )
    data class Text(
        val common: CommonProps,
        val content: PropValue,
        val typography: TokenRef,
        val color: TokenRef,
        val maxLines: Int? = null,
        val overflow: TextOverflow? = null,
        val textAlign: TextAlign? = null,
    )
    data class Fallback(
        val description: String,
        val level: FallbackLevel,
        val occurrenceCount: Int = 1,   // tracked for escalation
    )
}

sealed class SizeMode {
    object Fill : SizeMode()
    object Wrap : SizeMode()
    data class Fixed(val token: TokenRef) : SizeMode()
}
```

**`Direction.Stack`** = absolute positioning. Children have `offset` + `zIndex`, no mainAxis/crossAxis applies.

**TokenRef** = string key from `token-mapping.json`. Platform skill resolves:
- Android: `"spacing.medium"` → `16.dp`, `"color.primary"` → `colorResource(R.color.primary)`
- iOS: `"spacing.medium"` → `16` (CGFloat), `"color.primary"` → `Color("primary")`

**Platform extensibility:** Adding iOS = write one `resolveToken(TokenRef) -> SwiftUICode` + `renderLayout(Layout) -> SwiftUICode`. Core IR unchanged.

## 7. Token Resolution Algorithm

### Cache Layer (MUST check first)

At session start, build in-memory cache from `token-mapping.json`:
```
colorCache:    Map<Hex (lowercase), TokenEntry>
spacingCache:  Map<Dp (Int), TokenEntry>
typographyCache: Map<"fontSize:fontWeight:lineHeightBucket", TokenEntry>
```

All lookups hit cache first. Fallback algorithm runs only on cache miss.
Cache is immutable per session — no mid-session mutation. If `token-mapping.json` is updated, rebuild cache.

**Benefits:** deterministic output between runs on same input + significant speedup at scale.

### Colors (3-step lookup, cache-first)

```
0. Cache hit → emit token.compose immediately
1. Cache miss → grep colors_source (colors.xml or Color.kt)
   → found: emit colorResource(R.color.<name>)
             + auto-append to token-mapping.json + update cache
2. Not in colors_source → CIELAB closest match:
   ΔE < 2.0 → WARN "approximate match", use closest
   ΔE ≥ 2.0 → FAIL with actionable message

NEVER emit hardcoded Color(0x...) — always fail instead.
```

### Spacing (cache-first)

```
0. Cache hit → emit token.compose immediately
1. Cache miss → find candidates where |figma_dp - token.dp| ≤ 4
2. One candidate → WARN "approximate match Δ=Xdp", use token
3. Multiple with equal Δ → pick smallest dp (conservative), WARN about tie
4. No candidates within 4dp → FAIL
```

### Typography (cache-first)

```
0. Cache hit (fontSize:fontWeight:lineHeightBucket) → emit immediately
1. Cache miss → match ALL three: fontSize exact + fontWeight exact + lineHeight within 10%
   Partial match → FAIL with actionable message
```

## 8. Component Mapping Lifecycle

```
confidence=0.95  source=bootstrap  → AI exact name match
confidence=0.80  source=bootstrap  → AI fuzzy match
confidence=0.70  source=bootstrap  → AI semantic match
confidence=1.0   source=manual     → user confirmed
```

Confidence decay: -0.05/month unused (never below 0.3).
Consistency boost: +0.05 per correct use (cap 1.0).
Stale gate: confidence < 0.5 → flag for re-review. < 0.3 → treat as unmapped.

### Negative Learning

Each mapping entry stores rejected candidates:
```json
{
  "Button/Primary/Large": {
    "component": "AppButton",
    "confidence": 1.0,
    "rejected_candidates": ["TextButton", "OutlinedButton"],
    ...
  }
}
```

**Rules:**
- When user rejects a candidate (`n` in review-mappings), add to `rejected_candidates`
- AI matching MUST exclude rejected candidates in future runs — same Figma component, same context
- Rejected list is permanent (never auto-cleared). User can manually remove via `review-mappings --edit`
- Bootstrap scoring: if a candidate appears in `rejected_candidates`, score = 0 (excluded before threshold check)

**Effect:** System does not re-suggest `Button → TextButton` after user rejected it once. Prevents repeat noise.

## 9. FallbackLevel Rules

| Level | Condition | Behavior |
|---|---|---|
| M3 fallback | No custom mapping, but Material3 equivalent exists | Use M3 component, WARN |
| Minor | No mapping, no M3, at depth ≥ 3 or Icon/Badge/Chip/Avatar | WARN + emit Box with TODO |
| Major (nested) | No mapping, no M3, primary component, nested | Force staged mode, require user confirm |
| Major (root) | No mapping, no M3, primary component, root-level | BLOCK — do not generate |

### Fallback Escalation

System tracks fallback history in `.claude/figma-to-compose/fallback-history.json`:
```json
{
  "Badge/Status": { "count": 5, "last_seen": "2026-04-17", "level": "Minor" },
  "Card/Shift": { "count": 2, "last_seen": "2026-04-17", "level": "Major" }
}
```

**Escalation rules:**
- Minor fallback ≥ 3 occurrences → auto-suggest mapping via `suggest-mapping`
- Minor fallback ≥ 5 occurrences → upgrade to Major (force fix, same behavior as Major fallback)
- Major fallback ≥ 2 occurrences → escalate to BLOCK (treat as root-level regardless of depth)
- After `review-mappings` resolves → reset count to 0

**Effect:** System doesn't silently repeat the same fallback. Forces resolution.

## 10. Incremental Generation

Generated blocks are wrapped in markers:
```kotlin
// <figma-generated id="<stableId>" hash="<contentHash>" fingerprint="<fp>">
... generated code ...
// </figma-generated>
```

### Stable ID (nodeId-first)

1. `hash(figmaNodeId + semanticRole)` — survives rename, survives reorder
2. `"<ParentName>.<slotName>.<childType>"` — semantic path fallback
3. `"<ParentName>.children[<index>]"` — last resort

### Fingerprint (fallback when nodeId changes)

When figmaNodeId changes (designer deleted + recreated node), attempt fingerprint match before treating as new:

```
fingerprint = hash(
  text_content_sample,      // first 32 chars of text nodes
  layout_type,              // Direction + mainAxis + crossAxis
  child_count,
  relative_position_bucket  // top/middle/bottom within parent
)
```

Match algorithm on re-run:
1. Try nodeId match → found → incremental update
2. nodeId miss → try fingerprint match (fuzzy, threshold 0.8) → found → WARN "node likely recreated", update
   **Threshold MUST stay ≥ 0.8. Do NOT lower it — false positives (matching wrong node) are worse than missing a match.**
3. Both miss → treat as new node → append new block, old block becomes orphan

**Effect:** Designer duplicate/recreate scenarios don't generate duplicate blocks.

Developer edits OUTSIDE markers. Re-run updates INSIDE markers only. Orphan detection on `cleanup-orphans`.

## 15. Autonomy Evolution Phases

System evolves from manual to autonomous as confidence data accumulates:

| Phase | Trigger | Behavior |
|---|---|---|
| **Phase 1 — Manual** | Default (fresh install) | All mappings require user review. `needs_review: true` on everything. |
| **Phase 2 — Assisted** | ≥20 confirmed mappings, avg_confidence ≥ 0.85 | High-confidence (>0.90) mappings auto-accepted. User reviews <0.90 only. |
| **Phase 3 — Autonomous** | ≥50 confirmed mappings, avg_confidence ≥ 0.95, <2% regression in last 30 runs | Mappings auto-accepted IF confidence ≥ 0.90 AND usage_count ≥ 3. New/unused mappings (usage_count < 3) still require review regardless of confidence. |

**Phase stored in config.json:** `"autonomy_phase": 1`

**Regression detection:** If a previously confirmed mapping produces a Major fallback → revert phase by 1, alert user.

**Rule:** User can always force-override phase via config. Autonomy is earned, not assumed.

## 16. Long-Term Learning Loop

System improves systematically through three feedback channels:

### 1. Reject Memory (per-mapping)
Stored in `rejected_candidates` (see Section 8). Prevents repeat AI suggestions of known-bad matches.

### 2. Usage Pattern Learning
Track which Figma components appear together in the same file:
```json
"co_occurrence": {
  "Button/Primary": ["Card/Shift", "Header/Nav"],
  "Badge/Status": ["Card/Shift"]
}
```
High co-occurrence → boost confidence of paired mappings when context matches.

### 3. Context-Aware Confidence
Same Figma component in different contexts can have different correct mappings:
```
Button/Primary in "JobCard" context → AppButton (confidence 1.0)
Button/Primary in "Dialog" context  → DialogButton (confidence 0.85)
```
`context_key` in mapping entries: `"<ParentComponentName>/<FigmaComponentName>"`. Separate confidence per context.

### Learning Loop Cycle

```
Run → emit report → detect fallbacks/mismatches
   → fallback escalation (Section 9) triggers suggest-mapping
   → user resolves via review-mappings
   → mapping confirmed → confidence boost
   → next run: higher coverage, fewer fallbacks
   → phase progression (Section 15)
```

**Self-improving invariant:** Each resolved fallback makes the next run better. System cannot degrade unless designer adds new unmapped components.
